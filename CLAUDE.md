# taste.txt

## Product Philosophy

Build for demoability first.

Optimize for:

1. Clarity
2. Trust
3. Speed to value
4. Engineering depth
5. Beauty

Never optimize for novelty.

A user should understand:

* what the product does
* why it matters
* what is happening

within 60 seconds.

---

## Core Principle

Never hide the magic.

Users trust systems they can see working.

Prefer:

* streaming
* visible progress
* visible workflows
* visible tool usage
* visible reasoning summaries
* visible transformations

Avoid black-box experiences.

---

## AI

AI is an amplifier, not the product.

Good:

* AI + workflow
* AI + execution
* AI + automation
* AI + reasoning
* AI + assistance

Bad:

* AI wrappers
* AI everywhere
* random AI buttons
* AI as a substitute for engineering

If AI disappears, the product should still make sense.

---

## UX

Minimize decisions.

Prefer:

* opinionated flows
* smart defaults
* AI-assisted forms
* progressive disclosure
* visible actions
* obvious next steps

Avoid:

* configuration
* settings-heavy UX
* hidden functionality
* nested menus
* onboarding friction

Results should appear within 3-4 interactions.

---

## Forms

Every field must justify its existence.

Prefer:

* inferred inputs
* suggested values
* AI-generated values
* dynamic forms

Target:

* 0-3 required fields

Avoid:

* long forms
* setup screens
* configuration pages

---

## Information Architecture

Optimize for diameter, not feature count.

One powerful feature > five weak features.

Merge related functionality aggressively.

A product should feel smaller than it is.

---

## Engineering Visibility

Expose depth without exposing complexity.

Users see:

* workflows
* actions
* progress
* outcomes

Power users may inspect:

* logs
* traces
* reasoning
* execution paths

Never force technical complexity onto normal users.

---

## Design

Light mode only.

Prefer:

* flat design
* subtle depth
* clean outlines
* large typography
* structured spacing
* simple geometric layouts
* grid backgrounds
* strong hierarchy

Use:

* 2-3 colors maximum

Color is for:

* branding
* state
* emphasis

Avoid:

* gradients
* glassmorphism
* neon palettes
* cyberpunk aesthetics
* futuristic gimmicks
* excessive shadows

If it looks like a Dribbble concept, simplify it.

---

## Motion

Motion should explain.

Use:

* shimmer
* streaming
* subtle transitions
* progress indicators

Avoid:

* decorative animations
* excessive Framer Motion
* movement without meaning

Animation should communicate state.

---

## Cards

Use cards sparingly.

Avoid:

* card walls
* dashboard soup
* full-width card stacks

Prefer:

* direct layouts
* grouped information
* strong hierarchy

---

## Navigation

Prefer:

* fewer pages
* fewer actions
* fewer decisions

Every screen should have:

* one primary action
* one obvious path forward

Never make users hunt.

---

## Dashboards

Show only what changes decisions.

Prefer:

* 3 important metrics

Avoid:

* 20 irrelevant metrics
* fake analytics
* vanity dashboards

---

## Landing Pages

Be concrete.

Avoid:

* "revolutionary"
* "next-generation"
* "AI-powered"

Prefer:

* what it does
* how it works
* why it matters

Demonstrate value immediately.

---

## Demos

Demo-first is a product requirement.

Prefer:

* seeded data
* one-click success
* visible workflows
* immediate feedback
* fast wow moments

Avoid:

* empty states
* setup requirements
* manual configuration

A judge should understand the value within 30 seconds.

---

## Product Smells

Immediate rejection:

* gradients
* dark mode obsession
* cyberpunk aesthetics
* glassmorphism
* AI for everything
* hidden workflows
* configuration-heavy UX
* keyboard-dependent demos
* excessive animations
* feature bloat
* chat wrappers
* fake dashboards
* engineering hidden behind abstractions

---

## Definition of Good Taste

Maximum clarity.
Minimum complexity.

Strong engineering.
Visible intelligence.

Simple surface.
Deep systems.
